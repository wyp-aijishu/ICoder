- 搭建基于 ReAct 架构的 Agent 框架。使用工厂模式+策略模式实现 DeepSeek-v4 与 GLM-5.1 不同模型 API 热切换。通过 ToolRegistry 注册文件操作、代码检索、命令执行等7个内置工具，实现 llm 并行调用。
- 针对于上下文膨胀问题，基于动态 Token 预算设计上下文压缩算法。上下文会在 90% 最大上下文占用时自动触发摘要压缩，同时保留最近 3 轮完整消息不参与压缩。
- 设计索引 + 文件的项目记忆隔离储存结构。记忆支持显式命令储存与隐式提取，每轮对话结束开启线程异步提取用户、项目、纠正等信息注入记忆索引与记忆文件。记忆加载基于渐进式披露策略，只加载当前项目的记忆索引注入 system prompt。
- 实现网络检索与网页抓取功能，基于 trafilatura 提取网页正文转换为 .md 格式。并将 web_search 和 web_fetch 作为工具注册到 Agent 的工具链中，实现了 LLM 自主判断联网时机的智能工具选择。
- 基于JSON-RPC 协议与 stdio 传输方式，构建本地 MCP 服务端。针对 SPA 架构网页，接入 chrome-devtools-mcp，通过 take_snapshot 拿取 DOM 文本，覆盖微信公众号等 web_fetch 不可达场景。

## 实现细节

* llm的工厂+策略方法是如何设计的？
	* 首先定义策略接口，定义一个输入历史对话列表和tools，输出内容、tool call、token用量的chat模板方法。然后定义了一个所有llm继承的抽象类，实现了chat接口。并定义了url、name、apikey等所有llm共有属性。最后每一个llm实例继承这个抽象类，通过初始化不同的url、name、apikey来实现不同模型供应商的实现。
* 如何实现不同模型api切换？
	* agent实例内部有一个llm client，通过解析传入模型的名称，使用工厂方法创建对应实例，注入替换到agent中，实现切换。
* 工具是如何定义并注册的？
	* 每一个tool都包含了name、description、参数、callable执行方法。每个参数包含type和description，采用OpenAI function calling标准json schema，注册到统一工具列表，包含读取文件、写入文件、查询目录、执行命令、代码检索、网络搜索、网页爬取等7个内置工具和mcp 29个浏览器工具。
* 如何实现代码检索工具？
	* 代码检索分为两种模式：关键词匹配与正则表达式匹配。输入包括匹配模式、关键词或正则表达式，以及搜索目录。检索先递归遍历搜索目录，同时忽略默认目录比如.git，.venv等目录，在目录中遍历文件，忽略大于最大文件大小1M的以及二进制文件，逐行读取并将命中结果以{相对路径}:{行号}: {内容}格式返回。同时设置最长输出结果字符数量、最多输出结果数量，防止上下文过长。
* 如何实现工具并行调用？
	* llm返回tool call信息时，进行判断，单个工具直接执行，并以将结果与id配对返回上下文。多个工具时，开启线程池，最大线程数为3。每个线程包含工具调用信息，以及feture工具调用结果，构建线程信息列表。获取结果时依次遍历列表获取结果，同时设置最大执行时间60s，超时取消线程，并将超时信息作为结果保存。
* 短期记忆如何判断压缩时机？
	* 在每次与llm交互之前进行判断总token是否达到最大上下文的90%，或者使用/compact命令进行压缩。
* 短期记忆是如何压缩成摘要的？
	* 每次压缩保留三次user开头的对话，与之前的摘要内容，提取成新摘要加入系统提示词，包含用户意图与约束、已完成任务、关键决定、修改过的文件、未完成任务、错误及用户纠正等信息，同时重新计算token计数，并在历史上下文中移除压缩对话。
* 长期记忆如何储存加载的？
	* 每个项目的长期记忆存储目录以项目绝对路径的SHA256 哈希值命名。每个目录包含一个索引文件，和若干记忆文件，索引每一行对应一个记忆文件名称和摘要。每一个记忆文件分为四类：用户偏好、项目信息、纠正、外部资源，包含名称、摘要和详细内容。
	* 长期记忆只有在启动时一次加载索引到系统提示词，llm根据索引的摘要决定是否读记忆文件
* 长期记忆如何提取的？
	* 长期记忆提取有两种方式：使用save命令显示提取，在每轮对话后，判断是否有超过四轮未提取的内容，启动后台守护线程异步提取。
	* 提取内容以json形式储存，包括用户偏好、项目信息、纠正、外部资源四种类型、命名、描述和完整内容，如果没有值得提取的返回空json
* 如何解决后台提取间隔过短，导致同一时间更新索引文件？
	* 通过可重入锁实现对于索引文件的原子性写入，加载时全量加载更新后再覆盖，避免追加行导致索引错乱。
* 如何实现网络搜索？
	* 采用策略模式构建默认搜索引擎，根据query发起http请求，返回标题、url、摘要信息，并根据相同url进行去重，保留最多5条信息。
* 如何实现网页抓取？
	* 首先进行校验，要求http/https协议，且禁止访问localhost。下载采用流式读取，设置最大读取容量5M，防止内存溢出。正文提取采用 trafilatura ，能自动剥离导航栏、广告、侧边栏等噪音，只保留文章主体内容，输出 Markdown保留标题、列表、链接等结构，不包含图片。如果提取为空，则返回提示。
*  trafilatura提取页面原理？
	* 基于特征评分算法，首先根据标签，按照div、p、article等标签分割区块，每个区块内部计算正文文本密度、超链接密度、和区块长度综合打分，选取评分最高的区块作为正文。如果失败则进行算法降级。
* 如何实现mcp服务端？
	* 使用python 的mcp sdk。第一步构建stdio client，内部自动接入stdin和stdout。第二步创建session会话层，完成客户端发起握手请求，服务端返回结果。第三步调用list tool去发现工具，返回工具结果包装为tool注册到工具注册表中。
* 如何调用mcp工具的？
	* 和普通工具采用相同的工具调用逻辑，在运行工具时，需要传入mcp的server name、工具名和参数，在对应服务session中执行工具。session内部会构造json-rpc请求，通过stdout返回请求结果，并转为纯文本格式封装为tool结果。
* chrome-devtools-mcp主要用了哪些工具，如何注册的？
	* navigate_page	导航到指定 URL、select_page	切换到指定页面、take_snapshot	读取 DOM 文本快照（a11y tree）
## 小故事
* 手动打断
手动打断后对话出现400报错，原因是对话历史实时对话，部分tool call未产生result就被打断，导致工具调用与结果匹配不完整，传入llm会导致报错。采用解决方案是在每一轮用户对话开始前建立一个保存点，保存历史对话以及token用量，当触发打断抛出异常后，回滚检查点，抛弃这轮对话消息，防止出现toolcall污染。

* 记忆系统设计
长期记忆系统更新迭代了很多次，最初版本是将所有记忆内容存到一个json中，检索是按照jieba分词判断记忆内容是否包含关键字检索的，有两个问题：检索准确率低，完全检索不到语义关联。第二是所有项目记忆混在一起导致污染。于是第二版采用了milvus向量数据库存储记忆，做了一个类似于rag的检索系统，对于不同项目也根据绝对路径进行过滤。准确率提升了，但是引发了新问题：mlivus不支持windows，使用必须先启动docker，每次用户输入都需要去检索，每次记忆提取都需要做向量化，开销很大。最后更新到第三版，受到skill的启发，采用了索引+文件的策略，用绝对地址hash作为项目记忆存储目录，检索时只需要一次加载索引，按需查看记忆，减少了不必要的向量数据库开销。
* 上下文压缩
最开始上下文压缩没有考虑user边界，直接保留三轮llm交互，导致400报错，打断了tool call和tool的配对。
